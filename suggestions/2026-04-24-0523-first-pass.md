## Bugs

### [P0] Non-I-864 flow is hollow: chat, evidence, extract, greeting, form state are all still I-864

- **Where:** `/fill?formId=<anything>` (landed via `/start` results → "Start N-400")
- **Repro:** Run wizard as LPR + spouse-USC + goal=citizen → results include "Start N-400 →" → click, or open `http://localhost:3000/fill?formId=n-400`.
- **Expected:** Chat greets about N-400, evidence checklist is N-400-specific, `/api/chat` drives N-400 fields, PDF fields land in the right widgets.
- **Actual:** Header correctly flips to "Application for Naturalization / N-400" (proving `formId` is read), but:
  - Greeting bubble still says "I'll help you fill out your I-864" (`chat.greeting` key literally hardcodes "I-864" in `src/lib/i18n/messages.en.ts:57`).
  - Evidence checklist is `I864_EVIDENCE` unconditionally (`src/app/fill/page.tsx:453`).
  - `/api/chat` route has an I-864-only system prompt and I-864 `FIELD_META` / `SENSITIVE_PATHS` imports with no `formId` input (`src/app/api/chat/route.ts:5,34-102`). The client never sends `formId` to chat (`fill/page.tsx:362`).
  - `/api/extract` system prompt hardcodes I-864 field paths (`src/app/api/extract/route.ts`, top of SYSTEM_PROMPT).
  - PDF generation does route by `formId` (`/api/pdf/route.ts`), but the state it receives is still built by an I-864 chat loop using I-864 dotted paths, so non-I-864 PDFs will mostly be blank.
- **Fix hint:** Thread `formId` through `/api/case` state → client → `/api/chat` + `/api/extract`. Swap the system prompt + field schema + sensitive-path set based on `formId`. Gate `I864_EVIDENCE` behind a `EVIDENCE_BY_FORM[formId]` lookup. Replace `chat.greeting` with a form-aware key or interpolate `{formId}`.

## UX / wiring gaps

### `/start` results don't re-use language for the verdict label

- **Where:** `/start` results, any language
- **Issue:** The verdict pill text (`Likely eligible`, `Possibly eligible`, etc.) is sourced from a hardcoded map (`src/app/start/page.tsx:302-307`). The reasoning/evidence/summary text *is* localized by the LLM, but the pill stays English.
- **Impact:** Jarring in Spanish / Mandarin flows — the most prominent status signal is the only English word on a card full of correctly translated content.
- **Fix hint:** Drive pill label from an `eligibility.verdict.likely` key.

### `/start` header link "formcutter" brand points at `/fill`, not `/` — can't easily get back to the landing hero

- **Where:** `/start` (and `/fill`) header brand `<Link href="/fill">`
- **Issue:** Branding always routes to `/fill`; there's no way to reach the landing-marketing page from deep pages without editing the URL.
- **Impact:** Users who start the wizard and want to read trust copy / FAQs can't return easily.
- **Fix hint:** Point brand to `/`; add a separate "Open app" CTA.

### Case submit button enables at `filledCount >= 3` — trivially easy to submit an empty case to the reviewer

- **Where:** `/fill` submit button (`fill/page.tsx:609`)
- **Issue:** Three filled fields (e.g. family name + given name + one more) is enough to push a case to the reviewer queue. Combined with the bug where drafts leak into the rep queue, this is a noise vector for the demo too.
- **Impact:** Reviewer console floods with low-signal cases. Bad during a live demo.
- **Fix hint:** Gate submit behind the full `tier=required` set being non-empty, or at minimum `filledCount >= 10`.

### Evidence descriptions tell users the sponsor's domicile triggers the "Proof of U.S. domicile" row only if they enter a non-US country — with no visible hint

- **Where:** `/fill` evidence sidebar
- **Issue:** `domicile-proof` is conditionally shown (`evidence.ts:101-104`) based on `part4.countryOfDomicile` being present and non-US. Before the LLM captures a country, the item is invisible. Users living abroad have no way to see that they need to gather domicile proof until quite late in the chat.
- **Fix hint:** Either always render the item with a "(if you live abroad)" subtitle, or have the chat explicitly ask about domicile earlier.

## Feature suggestions

### Surface "Not yet supported" forms as coming-soon chips instead of hiding them

- **Context:** The `/start` results page already shows `{rawForm} · not yet supported` for forms outside `FORM_REGISTRY` (`start/page.tsx:430-438`). But the landing's `Forms we support` section only shows the supported set.
- **Suggestion:** On the landing, include a greyed-out row for `I-589`, `I-765`, `I-821`, `I-129F`, `I-751` etc. with a "coming soon" badge so eligibility-recommended paths don't surprise users when they land on the results page.
- **Why:** Reduces "why was I told about this form if you don't support it?" confusion.

### Preserve the eligibility result in `localStorage` so /fill can greet with the specific form context

- **Context:** Right now `/start` → click "Start N-400" drops all the wizard context. The `/fill` page doesn't know *why* the user is on N-400.
- **Suggestion:** Persist the last `WizardResult` + selected recommendation in `localStorage`; `/fill` reads it on mount and kicks off chat with "You told me you're an LPR married to a U.S. citizen aiming for the 3-year naturalization track…".
- **Why:** High trust signal, cheap to build, and wires `/start` to `/fill` into a continuous narrative instead of two disconnected tools.

### Warn when a tax return is uploaded without W-2s / schedules

- **Context:** `/api/extract` already collects `missingComponents` for tax returns (per its system prompt). Not surfaced anywhere in `/fill`.
- **Suggestion:** Inline warning on the upload card or evidence item tied to `extraction.missingComponents`.
- **Why:** #1 USCIS RFE cause per the project's own research. Low effort, huge credibility boost for the demo.

### Show rate-limit hints to users before they hit them

- **Context:** `/api/chat` limits 120/hr, `/api/simplify` 60/hr, `/api/extract` 30/hr per IP.
- **Suggestion:** Catch the 429 response in `/fill` and show a banner "You've sent a lot of messages — give it a few minutes" rather than silently erroring.
- **Why:** Current fail mode is "(error: rate_limited)" as a raw chat bubble (`fill/page.tsx:369`).

### Deep-link the eligibility-wizard state in the URL

- **Context:** Wizard state is in React state only; a refresh wipes it. Users on flaky connections will get frustrated.
- **Suggestion:** Encode answers in the query string or `localStorage` with TTL; resume where they left off.
- **Why:** Classic form-abandonment fix, negligible effort.
